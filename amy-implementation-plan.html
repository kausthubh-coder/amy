Expo Android plan

Amy calorie tracker implementation plan

Build a close Android recreation of the supplied Amy reference: dark onboarding, custom calorie targets, notes-style calorie logging, dictation, food photo capture, barcode lookup through Open Food Facts only, image-based label fallback, calorie-only widgets, and a preview APK.

1 Nutrition database: Open Food Facts only. No FatSecret, USDA, or dual lookup.
0 bio No birthday, gender, height, or activity onboarding. Use custom targets instead.
2 Android widgets: calories remaining, plus a mini Today widget with capture shortcuts.
APK Expo-managed Android build with EAS preview profile and installable artifact.

Reference Takeaways

From the teardown HTML plus the original and latest local screenshots.

Visual system

Dark, soft, rounded, high-contrast

Use near-black screens, large white headings, grey helper text, pill controls, purple primary actions, and dark cards with subtle borders. Keep cards at 8px radius in app surfaces where possible, while allowing native phone/sheet shapes to stay soft.

Onboarding

Short custom-goal setup

Keep the reference rhythm, progress bar, back button, mascot art, and sticky Continue button, but skip birthday, gender, height, and activity. Ask for calorie target, weight goal, uncertainty bias, and permissions.

Core loop

Notes first, forms second

Main screen starts as a blank note with "Start logging your meals..." and a bottom macro pill. Users type, speak, photograph, scan, or tap plus for saved meals. Tapping calories opens a larger nutrition modal; swiping Today moves across days.

Trust

Make uncertainty visible

Include calorie uncertainty preferences, result confidence, source badges, editable serving size, and a correction loop. Stats has two views: weekly macro charts and streak/calendar repair.

Explicit Scope

This is the product boundary for the first Android app.

Area Build Do not build Reason
Onboarding Welcome, custom calorie target, weight goal, calorie estimate bias, permissions, optional saved-meal setup. Birthday, gender, height, required activity level, or forced calculator setup. The user wants direct control: set the calorie number and weight goal without a long health questionnaire.
Barcode Scan barcode with Expo Camera, query Open Food Facts product data, then show confirmation. FatSecret, USDA, parallel provider selection, provider voting. Keeps the lookup source simple and matches the user's "Open Food Facts only" constraint.
Photo Use OpenRouter vision to estimate food from plate photos and return editable candidates. Menu scanning or restaurant menu OCR. Photo logging reduces friction without opening the much bigger menu-extraction problem.
Label fallback If barcode fails or product fields are missing, photograph nutrition label and parse it through OpenRouter vision. Writing product data back to Open Food Facts in MVP. Fast user result first; contribution flow can come later with account/auth decisions.
Voice Dedicated mic button for dictation plus normal keyboard dictation support. Voice assistant shortcuts in the first build. Dictation is core; lock-screen assistant automation can wait until logging is stable.
Today navigation Swipe left/right on Today to move to tomorrow or previous days. Tap the calorie pill to open an in-depth modal. A separate calendar-first daily navigation flow. Swiping keeps the main screen fast and close to the reference app.
Location Optional foreground location to add nearby restaurant/context hints to AI prompts. Background tracking, geofencing, menu discovery. Useful for calorie context but sensitive; keep it transparent and opt-in.
Widgets Build two Android widgets: calories remaining and mini Today with deep links to type, barcode, camera, mic, and saved meals. Water widgets, activity widgets, generic dashboard widgets. Widgets should shorten calorie logging, not broaden the product.
Water Nothing. All water tracking, goals, units, bottles, quick add, widgets. Requested removal; avoids diluting the calorie-tracking goal.

User Flow

The complete app loop from install to daily tracking.

Happy path

The app should make calorie tracking feel like writing a note, with smart capture modes tucked into the bottom control row.

1. Onboard Welcome, custom daily calorie target, weight goal, calorie uncertainty, capture-method education, permissions.
2. Start Today Land on the dark Today screen with empty note area, Today pill, streak/settings pill, and macro footer.
3. Capture food Type a meal, tap mic, take a food photo, scan a barcode, or tap plus to log a saved meal.
4. Estimate OpenRouter parses natural text/images. Barcode uses Open Food Facts only. Missing labels use OpenRouter OCR.
5. Confirm User edits calories, serving size, macros, and confidence before saving to the local daily log.
6. Review Tap calories for the detail modal, swipe to nearby days, and use Stats/Streaks for weekly review.

Technical Architecture

Keep it readable and Expo-friendly.

Stack

Expo + React Native + TypeScript

Use Expo Router for routes, React Native Reanimated for small transitions, Zustand for UI state, and Expo SQLite for durable local data. Use EAS for Android preview APKs.

Brand

Tough cartoony app icon

Use the generated dark/light icon concept as direction: confident sticker-like Amy mascot, near-black dark icon, off-white light icon, purple accent, orange calorie flame. Export adaptive Android foreground/background assets.

Widgets

Native Android widgets

Add an Expo config plugin that generates Android AppWidget providers and deep-link intents. Widget A shows calories remaining. Widget B is a mini Today capture panel.

Local data

SQLite as source of truth

Tables: user_profile, settings, food_entries, food_drafts, daily_summaries, saved_meals, widget_snapshots, corrections, streak_days, permissions_state, media_assets, off_cache.

Open Food Facts

Single product data provider

Implement src/services/openFoodFacts.ts with barcode lookup, custom User-Agent, rate-limit handling, local cache, and mapping from nutriments to calories, protein, carbs, and fat.

OpenRouter

AI parser and multimodal layer

Implement src/services/openRouter.ts for text meal parsing, photo estimation, label OCR, correction recalculation, and audio transcription. Keep the model configurable through env.

API Flows

No hidden second nutrition database.

Flow Client action Service call Saved result Error/fallback
Typed meal User types "two eggs and toast" in Today note. OpenRouter structured JSON prompt with custom targets, uncertainty bias, optional location context. Draft with items, calories, macros, confidence, reasoning, source = ai_text. Ask for portion details if confidence is low.
Saved meal User taps plus and chooses a saved meal. No network call. Confirmed entry copied from saved_meals into the selected day. If no saved meals exist, show create-from-current-entry and manual add options.
Dictation Mic button records speech or uses native speech recognizer. OpenRouter STT or native speech-to-text, then same typed meal parser. Transcript plus food draft; user can edit transcript before confirm. If STT fails, keep audio draft and offer retry/type.
Food photo User photographs a plate. OpenRouter vision prompt returns likely foods, portions, calories, macros, confidence. Photo draft with one or more editable food items. Ask user to add portion/brand details.
Barcode Expo Camera reads EAN/UPC barcode. Open Food Facts product endpoint only, using app User-Agent and fields for product name, brands, serving size, nutriments, image. Product draft with serving-size selector and source = open_food_facts. If missing or incomplete, show label-scan fallback.
Nutrition label User photographs package nutrition label. OpenRouter vision OCR extracts serving size, calories, macros, and confidence. Label draft with source = label_ocr and no external product provider. Ask for another photo if label is wrapped, blurry, or incomplete.
Correction User edits calories/macros after AI estimate. No network call required; optionally OpenRouter recalculates if user asks why. Correction row tied to original draft and final entry. Use corrections for local eval cases before prompt changes.

Expo Modules

Android-first capability list.

Camera

expo-camera

Food photos, nutrition label photos, and barcode scanning. Configure Android camera permission text and test barcode fill/framing on a real device.

Voice

Speech-to-text path

Preferred: @react-native-voice/voice in an EAS dev/preview build for native dictation. Backup: record audio with Expo AV and transcribe through OpenRouter STT when available for the selected model/provider.

Location

expo-location

Request foreground location only. Use it to suggest nearby restaurant context in AI prompts. Do not run background tracking or geofencing in MVP.

Notifications

expo-notifications

Optional meal reminders and streak nudges. On Android 13+, create a channel before requesting notification permission.

Storage

expo-sqlite + file system

Persist entries locally, cache Open Food Facts products, store thumbnail paths, and support JSON export/import.

Health

Android Health Connect later

The reference has Apple Health. For Android, defer Health Connect until the tracker loop is stable. It is not required for the first calorie-logging APK.

Build Phases

Order of implementation once coding starts.

Phase Deliverable Acceptance check
0. Scaffold Create Expo TypeScript app, Expo Router, lint/test setup, env template, SQLite migration helper, dark theme tokens. npm test and npx expo-doctor pass.
1. Visual shell Recreate short onboarding, top nav, Today shell, settings modal, two-tab stats modal, and generated app icon assets. Browser/native screenshots match reference spacing, contrast, dark mode, and light app-icon variant.
2. Local goals Custom daily calories, weight goal, macro targets, settings, export/import. Changing the custom target immediately updates Today, Stats, and widget snapshots after restart.
3. Text logging Notes-style input, OpenRouter structured parser, confirm/edit screen, saved daily entries, calorie-detail modal. User can log "two eggs and toast", edit calories, tap the calorie pill, and see daily totals update.
4. Barcode + Open Food Facts Camera barcode scanner, OFF client, product cache, serving-size confirmation. Known UPC scan creates an editable product draft using only Open Food Facts data.
5. Images Food photo estimation and nutrition label fallback through OpenRouter vision. Photo and label drafts include confidence and require user confirmation.
6. Dictation + location Mic flow, transcript editing, optional foreground location context. Spoken meal becomes editable text; location can be disabled with no broken flow.
7. Saved meals + day nav Plus-button saved meals, Today swipe to tomorrow/previous days, Stats/Streaks modal polish. Saved meals log in one tap and each day keeps independent notes/totals.
8. Android widgets Calories remaining widget and mini Today widget with deep links for type, barcode, camera, mic, and saved meals. Widgets update after logging and open the correct app surface from each tap target.
9. APK EAS preview Android build and local APK download. eas build --profile preview --platform android --non-interactive --wait produces installable APK.

Testing Plan

Concrete checks before calling the app shippable.

Unit

Nutrition math

Test custom calorie targets, macro target calculation, serving-size conversions, OFF nutriment mapping, day swiping totals, saved-meal copies, and correction diffs.

Integration

Provider contracts

Mock OpenRouter and Open Food Facts responses. Verify all AI flows return structured JSON and all OFF products map to editable drafts.

Device

Android permission checks

Test camera, microphone, notifications, photo library, and foreground location on the actual APK. Confirm denial states still leave typing/manual entry usable.

Visual

Screenshot regression

Capture onboarding, Today empty, keyboard-open, confirm, and streak screens. Compare them against this screenshot map for spacing, contrast, and bottom-safe-area behavior.

End goal

The first APK should make calorie tracking feel effortless: open the app, type or speak what you ate, scan packaged foods with Open Food Facts, photograph tricky foods or labels, confirm the estimate, and watch today's calories update. Everything else waits.
